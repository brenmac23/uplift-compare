  <name>Task 1: Create useFilteredProjects hook with filter logic and aggregate stats</name>
  <files>src/hooks/useFilteredProjects.ts</files>
  <action>
Create src/hooks/useFilteredProjects.ts with the following design:

**Types:**
```typescript
export type PassFailFilter = 'all' | 'existing-pass' | 'existing-fail' | 'proposed-pass' | 'proposed-fail' | 'both-pass';
export type TypeFilter = 'all' | 'film' | 'tv';
export type BudgetFilter = 'all' | 'under-100m' | 'over-100m';

export interface ScoredProject {
  project: Project;
  existing: ScoringResult;
  proposed: ScoringResult;
}

export interface AggregateStats {
  total: number;
  existingPass: number;
  proposedPass: number;
  bothPass: number;
}
```

**Hook signature:**
```typescript
export function useFilteredProjects(
  passFilter: PassFailFilter,
  typeFilter: TypeFilter,
  budgetFilter: BudgetFilter
): { filtered: ScoredProject[]; stats: AggregateStats }
```

**Implementation:**
1. Get projects from `useAppStore(s => s.projects)`
2. Use `useMemo` to compute scored list: map each project to `{ project, existing: scoreExisting(project.inputs), proposed: scoreProposed(project.inputs) }`
3. Use `useMemo` to compute filtered list from scored list applying all three filters:
   - PassFailFilter: filter on existing.passed / proposed.passed / both
   - TypeFilter: filter on project.inputs.productionType
   - BudgetFilter: filter on project.inputs.qnzpe >= 100_000_000 or < 100_000_000
4. Compute aggregate stats from the FILTERED list (not the full list) — this ensures stats update when filters change per CONTEXT.md locked decision
5. Return `{ filtered, stats }`

**IMPORTANT:** Stats must be derived from the filtered list, not the unfiltered list. See 03-RESEARCH.md Pitfall 7.
